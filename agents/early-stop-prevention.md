# 절차 4 — Early-Stop Prevention (조기 종료 방지)

fablize 실증 4절차 중 넷째다. 전이 가능 행동: 모델이 작업 중도에 멈추거나 범위를 임의로 축소하는 것을 감지·차단한다. 이 절차는 결정론 hook(`hooks/early-stop.sh`)으로 구현하며, 셀프테스트(`hooks/early-stop.test.sh`)로 검증한다. FL2 판정 기준이 "early-stop hook 셀프테스트"이므로 hook은 작성 후 직접 실행해 증거를 남긴다.

## 무엇

모델 출력에서 "X를 하겠다 / 할 것이다 / 다음에 하겠다"고 선언하고 실제로는 하지 않은 채 끝맺는 패턴을 잡는다. 대표 패턴은 다음과 같다.

- 미래 약속 후 종료: "이제 ~하겠습니다 / ~할 예정입니다 / 다음으로 ~하겠습니다"로 끝나고 그 행동의 증거가 없음.
- 범위 임의 축소: "나머지는 생략 / 일부만 / 시간 관계상 / 지면 관계상 / 여기까지만 / 이하 동일" 등으로 명시된 작업 범위를 줄임.
- 빈 완료: "완료했습니다 / 끝냈습니다"라고 하면서 절차 1의 증거가 동반되지 않음(evidence gate와 연동).

hook은 입력 텍스트(또는 파일)에서 이런 약속-미실행·범위 축소 패턴을 결정론으로 감지하면 종료 코드 비0을 반환한다. 패턴이 없으면 0을 반환한다. 판정은 산문 경고가 아니라 종료 코드로만 한다(NFR4).

## 적용 방법

1. 모델의 완료 직전 출력을 `hooks/early-stop.sh`에 텍스트 또는 파일 경로로 넘긴다.
2. hook이 비0을 반환하면 완료를 차단하고, 약속한 행동을 실제로 수행하거나 범위 축소 사유를 escalate한다.
3. hook이 0을 반환하면 evidence gate(절차 2)로 넘어간다.
4. hook 변경 시 `hooks/early-stop.test.sh`를 실행해 양성(차단되어야 할 케이스가 비0)·음성(통과해야 할 케이스가 0)을 모두 재검증한다.

hook은 macOS bash 3.2에서 안전해야 한다: 빈 배열은 `[ ${#arr[@]} -eq 0 ]`로 가드하고, `grep -c` 결과에 `|| echo 0`을 붙이지 않으며, 한글 출력 크래시를 피하기 위해 LC 설정을 고정한다.

## 왜

모델은 작업이 길어지면 "다음에 하겠다"는 약속으로 현재 턴을 끝내거나, 범위를 조용히 줄여 완료를 선언하는 경향이 있다. fablize 저자가 본 실패는 모델이 멀티스텝 작업의 후반부를 약속만 하고 실행하지 않는 것이다. early-stop hook은 이 패턴을 결정론으로 잡아 완료를 막는다.

이 절차는 모델에게 끈기라는 능력을 주입하는 것이 아니다(그건 전이 불가다). 출력의 약속-미실행 패턴을 외부 게이트로 감지해 모델이 그 턴 안에서 약속을 실행하도록 되돌린다. 따라서 전이 가능하고, capability 위조가 아니므로 FL9와 충돌하지 않는다.

## 언제 escalate

- 범위 축소가 정당한 외부 제약(FL15 승인 경계로 인한 차단, 실행 환경 부재) 때문이다. hook이 잡으면 우회하지 말고, 축소 사유를 escalate해 명시 보류로 표시한다.
- 약속한 행동이 모델 능력 밖이다(전이 불가 영역). 약속을 거짓 실행으로 메우지 않고 escalate한다.
- hook이 오탐(false positive)을 낸다(정상 완료를 약속-미실행으로 오인). 셀프테스트로 재현한 뒤 패턴 규칙을 보정한다. 단, fail-closed 방향(애매하면 차단)을 우선하고, 규칙 완화는 음성 케이스 재검증 후에만 한다.

조기 종료를 통과시키는 것보다, hook이 멈춰 세우고 escalate하는 것이 옳다. early-stop hook은 fail-closed로 동작한다.
