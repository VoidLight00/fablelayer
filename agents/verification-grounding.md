# 절차 1 — Verification Grounding (검증 grounding)

fablize 실증 4절차 중 첫째다. fablize 저자가 A/B 런으로 확인한 전이 가능 행동: 모델은 산출물을 직접 실행·관찰한 뒤에만 완료로 본다. capability(원시 추론력)는 전이되지 않지만, "실행하고 관찰한 뒤 완료한다"는 절차는 전이된다.

## 무엇

완료 주장마다 실측 증거를 첨부하도록 강제하는 절차다. "됐다 / 작동한다 / 통과 / 고쳤다 / 배포됨"은 다음 중 하나 이상을 직접 관찰한 결과를 첨부할 때만 보고한다.

- 명령 실행의 종료 코드(exit code)
- 테스트 출력(통과/실패 카운트)
- `grep` 매치 카운트
- 파일 존재(`ls`, `test -f`)
- 렌더 확인(스크린샷·HTML 출력·CLI 출력)

실행·관찰하지 않았으면 "미검증"이라고 표기한다. "적힌 대로 되겠지"는 증거가 아니다. 산문으로 "차단한다 / 막는다"만 적혀 있고 종료 코드가 없으면 SOFT 주장으로 간주하고 신뢰하지 않는다.

## 적용 방법

1. 완료 후보 산출물을 만든다.
2. 그 산출물을 실제로 실행하거나 관찰하는 명령을 정한다(빌드/테스트/lint/렌더).
3. 명령을 실행하고 종료 코드와 핵심 출력을 기록한다.
4. 기록한 증거를 완료 보고에 첨부한다. 증거 없는 완료 보고는 evidence gate(절차 2)에서 거부된다.

## 왜

fablize 저자의 결론은 모델 능력 자체는 하네스로 못 올린다는 것이다. 올릴 수 있는 것은 모델이 자기 능력 한계 안에서 덜 실수하게 만드는 절차다. 검증 grounding은 그 절차의 기반이다. 실행 없이 선언만 하면 모델은 자신이 만든 코드의 결함을 보고하지 못하고, 잘못된 완료가 누적된다. 실행·관찰을 강제하면 모델은 자기 출력의 실제 상태를 보게 되고, 그 시점에서만 신뢰할 수 있는 완료가 가능하다.

이 절차는 FableLayer가 생성하는 산출물의 품질 바닥을 만든다. capability를 위조하지 않으므로 FL9(성능 단언 차단)와 충돌하지 않는다.

## 언제 escalate

다음 경우 절차를 우회하지 말고 escalate(상위 판단·사람 개입 요청)한다.

- 산출물을 실행할 환경이 없다(빌드 인프라 부재, 외부 의존성 미설치). 가짜 실행 로그를 지어내지 않고 "실행 불가 — 미검증"으로 표시한다.
- 실행은 했으나 결과 해석이 모델의 도메인 지식을 넘어선다(전이 불가 영역). 추측으로 통과 판정하지 않고 escalate한다.
- 검증 명령 자체가 신뢰할 수 없다(테스트가 항상 통과하는 fail-open 구조). 검증 도구의 건전성부터 escalate한다.

escalate는 실패가 아니라 정직 신호다. 전이되지 않는 능력을 절차로 위장하는 것이 더 큰 결함이다.
