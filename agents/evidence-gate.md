# 절차 2 — Multi-Story Completion + Evidence Gate (완결성 evidence gate)

fablize 실증 4절차 중 둘째다. 전이 가능 행동: 여러 작업을 끝까지 완결하고, 근거 없는 "done"을 거부한다. 검증 grounding(절차 1)이 증거를 만드는 절차라면, evidence gate는 그 증거가 없는 완료 보고를 fail-closed로 막는 절차다.

## 무엇

완료 보고를 두 부분으로 분리한다.

- 선언(declaration) — "X를 했다"는 주장.
- 증거(evidence) — 절차 1이 만든 실측 결과(exit code / 테스트 출력 / grep 카운트 / 파일 존재 / 렌더 확인).

evidence gate는 선언만 있고 대응 증거가 없는 완료 보고를 거부한다. 거부는 산문 경고가 아니라 종료 코드 비0(fail-closed)으로 한다. 다중 작업(multi-story)일 때는 각 작업 단위마다 선언과 증거가 짝을 이루는지 검사한다. 한 작업이라도 증거가 비면 전체 완료를 거부한다.

multi-story completion은 "범위 안의 모든 작업을 끝낸다"를 의미한다. 작업 목록 중 일부만 끝내고 나머지를 조용히 누락하는 것은 early-stop(절차 4)과 evidence gate 양쪽 위반이다.

## 적용 방법

1. 완료할 작업 목록을 명시한다(각 작업 = 하나의 story).
2. 각 작업마다 절차 1의 검증 명령과 그 종료 코드·출력을 수집한다.
3. 작업↔증거 매핑을 만든다. 매핑이 빈 작업이 하나라도 있으면 fail-closed로 거부한다.
4. 모든 작업에 증거가 있을 때만 완료로 본다.

evidence gate는 OR 집계로 판정한다. 한 작업이라도 증거 누락·검증 실패면 전체 결과는 실패다. 일부 통과를 전체 통과로 보고하지 않는다.

## 왜

모델은 그럴듯한 완료 보고를 만드는 데 능하지만, 그 보고가 실제 상태와 일치하는지는 별개다. fablize 저자가 본 실패는 모델이 "전부 끝냈습니다"라고 보고하면서 실제로는 일부만 했거나 검증하지 않은 경우다. evidence gate는 이 격차를 코드로 닫는다.

선언≠증거 원칙을 게이트로 강제하면, 모델이 자기 보고의 진실성을 스스로 점검하게 된다. 이는 capability 주입이 아니라 보고 규율 주입이다. 따라서 전이 가능하고, FL9와 충돌하지 않는다.

## 언제 escalate

- 작업의 완료 기준이 모호해 증거를 정의할 수 없다(예: "코드를 더 좋게 만들어라"). 검증 가능한 완료 기준부터 escalate해 합의한다.
- 검증이 모델 능력 밖의 판단을 요구한다(전문 도메인의 정합성 판정). 증거를 위조하지 않고 escalate한다.
- 다중 작업 중 일부가 외부 승인·차단(FL15 같은 승인 경계)으로 진행 불가하다. 그 작업은 "보류 — 승인 대기"로 표시하고, 나머지를 완료로 위장하지 않는다.

증거를 만들 수 없는 작업을 통과시키는 것보다, escalate로 게이트를 멈추는 것이 옳다. fail-closed가 기본값이다.
