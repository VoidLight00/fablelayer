# Smart Model Router — 규칙 (Layer 4 / FL4)

> 기본 모델은 Sonnet이다. 작업이 복잡해질 때만 Opus로 자동 전환한다. 전환 결정은 *문서화된 명시 기준*으로
> 내린다 — 라우팅이 암묵적이면 가성비(FL3)와 품질(FL16)이 둘 다 검증 불가가 된다.
>
> 이 파일이 FableLayer 모델 라우팅의 단일 출처다. Layer 3(ValueOptimizer)의 COST optimizer는 같은 신호·같은
> 분기를 쓰도록 정렬한다. 두 곳이 갈라지면 어느 한쪽이 조용히 무시되므로, 충돌 시 conductor가 우선순위를 정한다.

## 기본 규칙

1. **기본값 = Sonnet.** 신호가 하나도 켜지지 않으면 Sonnet으로 처리한다.
2. **상향 전환(escalate) = Sonnet → Opus.** 아래 "Opus 전환 신호" 중 하나라도 켜지면 Opus로 보낸다. 신호는 검증 가능한 관찰 가능 조건으로만 한정한다.
3. **하향 없음 정책.** Opus로 시작해야 하는 명시 작업(아키텍처 결정·전체 하네스 감사)은 신호와 무관하게 Opus를 유지한다. 비용 절감을 이유로 깊은 추론 작업을 Sonnet으로 강등하지 않는다.
4. **전이 불가 능력은 정직 표시.** 라우팅으로 옮기는 것은 *작업의 배치*이지 모델의 capability가 아니다. 어떤 라우팅도 Sonnet에 Opus의 추론력을 주입하지 못한다. capability가 결과를 좌우하는 작업은 신호 없이도 Opus로 escalate하고, 그 사실을 보고에 명시한다.

## Opus 전환 신호 (하나라도 켜지면 escalate)

| 신호 | 관찰 가능 조건 (이진 판정) | 근거 |
|------|---------------------------|------|
| 다단계 의존 작업 | 한 단계의 출력이 다음 단계 입력이 되는 의존 체인이 3단계 이상 | 중간 오류가 누적되면 평이 모델에서 드리프트 |
| 깊은 추론 | 아키텍처 결정 / 디버깅(원인 추적) / 인과 분석 / 트레이드오프 비교가 작업의 핵심 | 중립 재검증상 깊은 추론은 Opus 우세(아래 한계 절) |
| 다파일 리팩토링 | 변경 대상 파일이 3개 이상이거나 경계면(인터페이스) 동시 수정 포함 | 경계면 정합은 전역 추론을 요구 |
| 긴 컨텍스트 진입 | 컨텍스트 사용이 마지막 20% 구간 진입(대규모 멀티파일 작업 중) | 컨텍스트 후반 성능 저하 위험 구간 |
| 게이트 반복 실패 | 같은 게이트가 2회 연속 실패해 원인 가설 경쟁이 필요 | systematic investigation(FL2)이 깊은 추론 작업으로 승격 |
| 명시적 capability 의존 | 작업 정의가 "설계/감사/아키텍처/근본원인"을 핵심으로 지정 | capability가 결과를 좌우 — 신호 없이도 escalate |

신호가 켜지지 않는 작업(정형 변환, 단일 파일 편집, 스캐폴드 생성, 목록·표 작성, 패턴 적용)은 Sonnet으로 처리한다. value-for-fable 중립 재검증상 이 범위에서 Sonnet+구조는 노이즈 내 동률이며 단가가 낮다.

## 비용·품질 트레이드오프

- Sonnet은 평이·정형 작업에서 구조를 갖추면 Opus와 **노이즈 내 동률**로 관찰되었고 단가가 낮다(중립 재검증 범위). 이 구간을 일괄 Opus로 처리하면 가성비 부채가 생긴다(FL3).
- Opus는 깊은 추론에서 우세하다. 전환 신호가 켜진 작업을 Sonnet으로 처리하면 품질 부채가 생긴다.
- 라우터의 목적은 *작업마다 알맞은 모델로 보내는 것*이다. 토큰 절감은 라우팅으로 달성하며, 출력 압축으로 달성하지 않는다 — "압축/compress" 강제 규칙은 도입하지 않는다(FL3, value-for-fable v2 교훈: 압축은 품질 부채).

## 근거 한계 (정직성)

- "Sonnet+구조 ≈ Opus"는 중립 재검증에서 *노이즈 내 동률*로 관찰된 범위다. value-for-fable 원판 벤치는 재현에 실패했고(REQUIREMENTS.md 근거 절), 우리 결론은 그 재검증 범위로 한정된다.
- 깊은 추론에서는 Opus가 우세하다는 단서가 관찰되었다. 위 전환 신호의 "깊은 추론" 분기는 이 단서에 근거한다.
- 정량 우위(예: "N점", "M% 절감")는 이 문서에서 단언하지 않는다. 모든 수치 주장은 `bench/RESULTS.md` 실측과 `## 한계` 섹션 참조로만 한다(FL9, `gates/perf_claim_gate.sh`가 미검증 단언을 HARD 차단). 라우팅 기준은 *관찰된 우세/동률의 방향*에 근거하며, 특정 점수를 약속하지 않는다.
- 작업 분류(어떤 신호가 켜졌는지)는 휴리스틱이다. 경계 사례는 안전하게 Opus로 escalate하고, 그 판단을 보고에 남긴다.

## Layer 3 COST optimizer와의 정렬

- 이 router.md의 전환 신호 표가 라우팅의 단일 출처다. Layer 3(`styles/vff-v2.md`)의 COST 라우팅 규칙은 같은 신호 목록·같은 이진 조건을 참조한다.
- 두 문서가 갈라지면(예: Layer 3가 다른 임계값을 쓰면) 어느 한쪽이 조용히 무시된다. 불일치를 발견하면 임의로 한쪽을 따르지 말고 conductor에 보고해 우선 규칙을 받는다.

## 보고 규약

- 모든 라우팅 결정은 *어떤 신호로 어느 모델을 골랐는지*를 함께 기록한다(`runs/<run_id>/RUN_MANIFEST.json` 또는 _workspace 로그).
- capability 의존으로 escalate한 경우 "절차는 전이되나 capability는 전이 불가 → Opus로 처리"를 명시한다(전이 불가 능력의 정직 표시, REQUIREMENTS.md 정체성 절).
- 라우터의 효과 판정은 서술이 아니라 `bench/`의 비용·품질 실측으로 한다(자가보고 금지, NFR4).
