# COST — 작업유형별 모델 라우팅 기준

이 문서는 FableLayer ValueOptimizer(Layer 3, FL3)의 비용 라우팅 기준이다. 비용 절감은 출력 압축이 아니라 **모델 라우팅**으로 달성한다. 기본 모델은 Sonnet이고, 깊은 추론이 필요한 작업만 Opus로 승격한다. 이 표는 Smart Model Router(Layer 4)가 소비하는 입력 형식이다.

## 큰 분류: 입력형 vs 출력형

라우팅의 1차 기준은 작업이 "긴 입력을 읽고 좁은 답을 내는가(입력형)"인지, "짧은 입력에서 긴 산출을 만드는가(출력형)"인지다. 단가 구조가 입력 토큰과 출력 토큰에서 다르기 때문에 이 구분이 라우팅의 출발점이다.

| 축 | 입력형 작업 | 출력형 작업 |
|---|---|---|
| 성격 | 큰 컨텍스트(코드베이스·로그·문서)를 읽고 좁은 결론/패치를 낸다 | 짧은 사양에서 긴 산문/문서/카피/설명을 생성한다 |
| 대표 예 | 코드 리뷰, 버그 grep, 로그 분석, 구조 추출, 분류, 추출 | 글쓰기, 문서 작성, README, 카피, 설명문, 요약본 |
| 비용 지배 항목 | 입력 토큰(컨텍스트가 큼) | 출력 토큰(생성이 김) |
| 기본 모델 | Sonnet | Sonnet |
| Opus 승격 조건 | 다단계 인과 추론·아키텍처 판단이 섞일 때만 | 깊은 논증·전략·창의적 구조 설계가 핵심일 때만 |

두 유형 모두 **기본은 Sonnet**이다. Opus 승격은 유형이 아니라 "추론 깊이"가 결정한다(아래 승격 트리거).

## 작업유형별 기본 라우팅

| 작업유형 | 유형(입력/출력) | 기본 모델 | 비고 |
|---|---|---|---|
| 코드 리뷰·diff 검토 | 입력형 | Sonnet | 구조화 출력 스타일과 결합 시 충분 |
| 버그 재현·로그/grep 조사 | 입력형 | Sonnet | 인과 사슬이 길어지면 Opus 승격 |
| 코드베이스 구조 추출·매핑 | 입력형 | Sonnet | |
| 분류·추출·정규화 | 입력형 | Sonnet | 결정론에 가까운 작업 |
| 문서·README·가이드 작성 | 출력형 | Sonnet | VFF v2 구조 적용 |
| 카피·설명문·번역 | 출력형 | Sonnet | |
| 단발 변환·포맷팅 | 입력/출력 단순 | Sonnet | 진단형 구조 비례 적용(최소형) |
| 아키텍처 결정·시스템 설계 | 추론 지배 | **Opus** | 다단계 트레이드오프 판단 |
| 다단계 인과 추론·근본원인 분석 | 추론 지배 | **Opus** | 가설 경쟁이 깊을 때 |
| 어려운 알고리즘·고난도 디버깅 | 추론 지배 | **Opus** | |
| 전략·연구·설계 논증 | 추론 지배 | **Opus** | |

## Opus 승격 트리거 (Sonnet → Opus)

다음 중 하나라도 해당하면 Sonnet 기본에서 Opus로 승격한다. 해당하지 않으면 Sonnet을 유지한다.

1. **다단계 인과 추론** — 원인 → 중간 효과 → 결과가 3단계 이상 얽혀 한 단계 오판이 전체를 무너뜨리는 작업.
2. **아키텍처/설계 결정** — 되돌리기 어렵고 트레이드오프가 여러 축인 구조 결정.
3. **경쟁 가설이 깊은 조사** — 단순 grep으로 좁혀지지 않고, 여러 가설을 각각 검증해야 하는 근본원인 분석.
4. **고난도 알고리즘·정합성** — 정확성 증명·복잡한 불변식·동시성 등 미묘한 추론이 필요한 작업.
5. **Sonnet 1차 시도 실패** — Sonnet으로 시도했으나 구조화 출력에도 결론이 흔들리거나 검증을 통과하지 못한 경우(에스컬레이션).

승격은 작업 단위로 한다. 한 세션 안에서도 입력형 정리 작업은 Sonnet, 그 위의 설계 판단은 Opus로 나눌 수 있다.

## 단가·동률에 대한 정직한 표기 (FL9 준수)

라우팅의 전제가 되는 비용/품질 관계는 다음과 같이 **단언이 아니라 한계 명시와 함께** 기록한다. 수치 성능 주장은 `bench/` 실측 참조 또는 한계 명시 없이 쓰지 않는다.

- value-for-fable 중립 재검증 범위에서 관찰된 경향: 구조화 출력 스타일을 적용한 Sonnet은 다수 일상 작업에서 Opus와 노이즈 범위 내 동률에 가까웠고, 토큰 단가는 약 0.30배 수준이었다. **이는 그 재검증 범위 내 관찰이며 모든 작업에 일반화되지 않는다(한계).** 깊은 추론·다단계 인과 추론 작업에서는 Opus가 우세했다 — 이 영역까지 Sonnet으로 내리면 품질이 떨어진다.
- 이 문서의 단가·동률 진술은 FableLayer 자체 벤치 실측(`bench/RESULTS.md` 및 그 `## 한계` 섹션)으로만 확정한다. 벤치 참조가 없는 수치는 "추정"으로 표기하거나 적지 않는다.
- 비용 절감 효과를 숫자로 단언하지 않는다. 절감은 "Sonnet 기본 + 선택적 Opus 승격" 구조에서 **작업 분포에 따라** 달라지며, 고정 절감률을 약속하지 않는다(한계).

## 라우팅이 하지 않는 것

- 출력 압축으로 비용을 줄이지 않는다. 압축 규칙은 품질 부채다(FL3). 절감은 모델 선택으로만 한다.
- 비용을 이유로 추론 지배 작업을 Sonnet에 강제로 묶지 않는다. 승격 트리거에 해당하면 Opus로 올린다 — 잘못된 결론의 재작업 비용이 단가 차이를 압도한다.
- capability를 라우팅으로 이식하지 않는다. 라우팅은 적합한 모델을 고르는 것이지, 약한 모델을 강하게 만드는 것이 아니다.
